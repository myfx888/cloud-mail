## 上下文

当前写信弹窗中的回复模板和转发模板由 `mail-vue/src/layout/write/index.vue` 直接拼接 HTML。回复模板包含 `formatDetailDate(email.createTime)` 与 `t('wrote')`，因此会随 `settingStore.lang` 在中英文之间切换；转发模板目前仅嵌入原始内容，缺少标准转发头。用户期望“回复/转发元信息固定英文、正文保持原语言”，这要求在局部场景覆盖全局 i18n 行为。

## 目标 / 非目标

**目标：**
- 在回复邮件中固定英文元信息（英文日期 + `wrote:`）。
- 在转发邮件中输出标准英文头（`Forwarded message`、`From`、`Date`、`Subject`、`To`）。
- 主题前缀统一为英文策略（`Re:` / `Fwd:`）并避免重复前缀。
- 不改变引用正文内容，保持原始 HTML/纯文本内容不被翻译或重写。

**非目标：**
- 不修改全局语言设置与其他页面的国际化行为。
- 不改造服务端邮件存储与收发协议逻辑。
- 不引入新的邮件模板引擎。

## 决策

1. **在回复/转发路径引入“英文专用元信息格式化”能力。**
   - 做法：在日期工具中新增英文固定格式函数，或在写信页内使用 `dayjs` 固定英文 locale 与格式。
   - 理由：局部覆盖可满足需求，同时避免影响全局 `settingStore.lang` 的既有行为。
   - 备选：直接复用 `formatDetailDate` 并临时切换全局语言；放弃，因为会产生全局副作用与竞态风险。

2. **回复模板不再依赖 `t('wrote')`，改为固定英文关键字。**
   - 做法：回复引用头始终使用 `wrote:`。
   - 理由：该字段属于邮件元信息，不应随 UI 语言变化。
   - 备选：为 `t()` 增加“强制某语种”参数；放弃，因为会扩散 i18n 接口复杂度。

3. **转发模板补全标准英文头并与正文分离。**
   - 做法：在 `openForward` 生成如下结构：转发分隔行 + `From/Date/Subject/To` + 原正文块。
   - 理由：满足“特别是转发邮件时”的可读性与跨语言一致性目标。
   - 备选：保持当前仅正文转发；放弃，因为无法实现“元信息默认英文”。

4. **主题前缀去重采用中英混合识别，输出统一英文。**
   - 做法：识别现有 `Re:`/`回复:`、`Fwd:`/`Fw:`/`转发:` 等前缀，若缺失则补英文前缀。
   - 理由：兼容历史邮件主题，同时保证新生成主题统一英文。

## 风险 / 权衡

- **[风险] 不同客户端对转发头样式容忍差异** → **缓解措施：** 采用最常见英文头字段与简单 HTML 结构，避免复杂样式。
- **[风险] 前缀去重规则遗漏变体** → **缓解措施：** 明确覆盖常见中英文前缀并补充回归用例。
- **[权衡] 局部硬编码英文文本** → **说明：** 与“元信息固定英文”目标一致，且范围受限于回复/转发生成逻辑。

## 迁移计划

- 第一步：新增英文元信息格式化辅助函数并在写信页接入。
- 第二步：调整 `openReply`、`openForward` 模板与主题前缀逻辑。
- 第三步：手工验证中英文界面下回复/转发结果，确认正文不变。
- 回滚策略：若出现兼容性问题，可回退至旧模板（仅影响写信页前端逻辑，无数据库迁移）。

## 待确认问题

- 转发头中的 `To` 字段在原邮件缺失收件人时，是否显示为空行或整行隐藏。
- 转发分隔行文案是否固定为 `---------- Forwarded message ---------`。
