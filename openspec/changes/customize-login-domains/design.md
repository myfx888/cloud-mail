## 上下文

目前，Cloud Mail 的域名后缀是直接从环境变量 `domain`（JSON 数组）中读取并暴露给前端的。这导致所有配置的域名都会在登录页和注册页显示，缺乏灵活性。

## 目标 / 非目标

**目标：**
- 在数据库中持久化存储“可见域名”配置。
- 提供管理界面供管理员勾选显示的域名。
- 后端 API 能够根据配置动态过滤返回给前端的域名列表。
- 确保在未配置时（默认状态）显示所有域名，保证兼容性。

**非目标：**
- 修改域名解析或邮件投递逻辑（这些仍由环境变量和 Mailcow 配置决定）。
- 改变用户已绑定的域名归属。

## 决策

### 1. 数据库存储方案
- **选择**：在 `setting` 表中增加 `login_domains` 字段，类型为 `TEXT`，存储以逗号分隔的域名字符串。
- **理由**：现有的 `setting` 表已存储了大量的全局配置，增加字段是最简单的扩展方式。逗号分隔字符串比 JSON 在 SQLite 中更易于简单处理，且域名数量通常不多（< 20个）。

### 2. 后端过滤机制
- **位置**：`settingService.websiteConfig` 方法。
- **逻辑**：
  1. 获取环境变量 `c.env.domain`。
  2. 获取数据库 `setting.login_domains`。
  3. 如果 `login_domains` 为空或未设置，返回全部。
  4. 否则，返回 `c.env.domain` 与 `login_domains` 的交集。
- **理由**：`websiteConfig` 是前端初始化的核心接口，在这里过滤可以确保全局生效（Store 层）。

### 3. 前端 UI 组件
- **选择**：在系统设置中使用 `el-checkbox-group`。
- **理由**：域名是预定义的列表（来自 `env.domain`），多选框是最直观的交互方式。

## 风险 / 权衡

- **[风险]** 管理员删除了环境变量中的某个域名，但数据库中仍勾选。
  - **缓解**：过滤逻辑应以 `env.domain` 为基准，数据库仅作为“显示过滤器”。
- **[风险]** 管理员取消勾选所有域名。
  - **缓解**：代码逻辑处理为“若过滤器结果为空，则显示全部”，防止前端列表出现空白。
- **[权衡]** 存储格式。
  - **决策**：虽然 `setting` 表有些字段用了 JSON，但为了保持与 `email_prefix_filter` 等字段的一致性，采用逗号分隔。
