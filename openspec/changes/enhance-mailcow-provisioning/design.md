## 上下文

### 当前状态

现有 Mailcow 集成已实现基础功能：
- 后台可配置多 Mailcow 服务器（`mailcowServers` JSON 数组）
- 新增邮箱时自动调用 Mailcow API 创建账户
- 使用随机密码生成（`mailcowService.generatePassword()`）
- SMTP 配置从服务器配置读取（`server.smtpHost` 等）
- 失败时不阻断本地账户创建（宽松模式）

### 约束

- 数据库为 SQLite（Cloudflare D1），字段变更需迁移脚本
- Mailcow API 使用 API Key 认证，无用户级权限隔离
- 前端使用 Vue 3 + Element Plus
- 敏感字段需脱敏处理，避免日志泄露

### 利益相关者

- 系统管理员：配置 Mailcow 参数、统一密码、失败策略
- 普通用户：无感知，自动获得 SMTP 配置

## 目标 / 非目标

**目标：**

1. 实现双密码模式（`fixed` / `random`），支持统一密码开通
2. 实现创建失败策略开关，支持严格/宽松模式切换
3. 实现全局 SMTP 模板，作为服务器配置的 fallback
4. 实现重复账户检测，已存在邮箱视为成功
5. 实现敏感字段脱敏，保护 API Key 和统一密码
6. 为 Mailcow 配置失败项提供可视化重试按钮
7. 实现 SMTP 多服务器配置与账户级选择
8. 统一账户界面 SMTP 相关重复按钮为单一入口

**非目标：**

1. 不实现用户级密码自定义（统一密码由管理员配置）
2. 不实现 Mailcow 多租户隔离（使用单一 API Key）
3. 不实现密码加密存储（明文存储，依赖数据库访问控制）
4. 不实现 Mailcow 账户同步删除（仅创建流程增强）

## 决策

### 决策 1：密码模式存储方式

**选择：** 明文存储 `mailcowProvisionPassword`，查询时脱敏

**理由：**
- Mailcow API 需要明文密码创建账户
- 数据库已有访问控制，无需额外加密层
- 脱敏在 API 层处理，避免日志泄露

**替代方案：**
- AES 加密存储：增加复杂度，密钥管理困难
- 环境变量存储：无法通过 UI 配置，运维不友好

### 决策 2：重复账户检测策略

**选择：** 调用 Mailcow API 前先查询邮箱是否存在，存在则跳过创建

**理由：**
- 减少 API 调用次数
- 避免 Mailcow 返回错误时的歧义处理
- 已存在邮箱仍需回填 SMTP，流程一致

**替代方案：**
- 直接创建，捕获错误后判断：可能导致不必要的失败日志
- 不处理重复：用户体验差，需手动清理

### 决策 3：SMTP 配置来源优先级

**选择：** 服务器配置优先，全局模板 fallback

**优先级顺序：**
1. `mailcowServers[i].smtpHost/smtpPort/smtpSecure/smtpAuthType`
2. `mailcowGlobalSmtpTemplate.smtpHost/smtpPort/smtpSecure/smtpAuthType`
3. 硬编码默认值（`smtp.mailcow.email:587`）

**理由：**
- 不同 Mailcow 服务器可能有不同 SMTP 配置
- 全局模板减少重复配置
- 硬编码默认值保证兜底

### 决策 4：失败策略实现位置

**选择：** 在 `account-service.js` 的 `add` 方法中实现

**理由：**
- 账户创建是主流程，Mailcow 创建是副作用
- 失败策略影响是否回滚本地账户
- 集中在 `account-service` 便于维护

**替代方案：**
- 在 `mailcow-service.js` 抛出不同错误类型：需修改错误处理架构
- 在 API 路由层处理：职责分散，不易维护

### 决策 5：敏感字段脱敏方式

**选择：** 后端 API 返回时替换为 `******`，前端不处理

**脱敏规则：**
- `mailcowProvisionPassword`: 完全隐藏（`******`）
- `mailcowServers[].apiKey`: 部分隐藏（`key****xxx`，显示前4后3字符）

**理由：**
- 后端统一处理，前端无需感知
- 避免敏感数据传输到客户端
- 日志中也不会出现明文

### 决策 6：失败项重试入口与执行方式

**选择：** 在账户列表中对 `mailcowStatus=failed` 的记录显示“重试”按钮，点击后调用独立重试接口。

**执行规则：**
1. 仅失败状态显示重试按钮，进行中状态禁止重复点击
2. 点击后调用 `POST /api/account/:accountId/mailcow/retry`
3. 重试逻辑复用首次创建流程（重复检测、密码模式、SMTP 合并、失败策略）
4. 重试完成后刷新该账号状态与错误信息

**理由：**
- 失败项在账户维度发生，入口放在账户列表更直观
- 独立接口便于权限控制和审计
- 复用主流程可降低分叉逻辑风险

**替代方案：**
- 在系统设置页统一“全部重试”：缺乏按账号精确控制
- 自动后台重试：容易引发风暴式重试与难以感知的副作用

### 决策 7：SMTP 多服务器数据模型与选择策略

**选择：** 在系统设置中维护 `smtpServers` 列表，账户侧存储 `smtpServerId` 作为绑定关系。

**执行规则：**
1. `smtpServers` 支持新增、编辑、禁用、设置默认服务器
2. 新建账户时，默认绑定 `smtpServers` 中 `isDefault=true` 的服务器
3. 账户支持手动切换 `smtpServerId`，切换后立即重算并写入 SMTP 字段
4. 当绑定服务器被禁用时，账户侧提示并要求选择新的可用服务器

**理由：**
- SMTP 是可复用资源，列表化管理优于每账号重复维护
- 通过 `smtpServerId` 绑定便于追踪与批量治理
- 默认服务器可降低新建账户配置成本

**替代方案：**
- 每个账户独立维护 SMTP：配置重复且容易漂移
- 每个用户只允许一个全局 SMTP：无法满足多环境/多域需求

### 决策 8：Mailcow 多服务器路由策略

**选择：** Mailcow 服务器维持多实例配置，创建/重试时按账号绑定优先，未绑定则使用默认服务器。

**执行规则：**
1. 系统设置允许管理多个 `mailcowServers`，且必须保证最多一个默认服务器
2. 账户可记录 `mailcowServerId`，优先使用该服务器执行 API 操作
3. 未绑定 `mailcowServerId` 时，回退到默认 Mailcow 服务器
4. 当目标服务器不可用时，按重试策略返回失败，不自动跨服务器漂移

**理由：**
- 账号与 Mailcow 实例绑定可避免跨实例数据不一致
- 默认服务器支持平滑接入，不影响存量流程
- 禁止自动漂移可避免在错误实例误创建账号

### 决策 9：账户界面 SMTP 按钮统一

**选择：** 账户界面所有 SMTP 相关操作统一为一个“SMTP 设置”入口，内部提供页签/分组完成原有操作。

**执行规则：**
1. 列表行内仅保留一个 SMTP 操作按钮，移除重复按钮
2. 统一弹窗包含：服务器选择、参数预览、测试连接
3. 对失败状态与重试入口保持同一视觉层级，避免操作分散

**理由：**
- 减少重复操作入口造成的认知负担
- 统一交互路径便于后续扩展多服务器能力

### 决策 10：回复编辑窗口 SMTP 切换交互

**选择：** 在邮件回复编辑窗口右下角提供 SMTP 服务器下拉框，作为当前回复会话的发送服务器选择器。

**执行规则：**
1. 仅当可用 SMTP 服务器数量大于 1 时显示下拉框
2. 默认选中账户当前绑定的 `smtpServerId`，未绑定时选中默认 SMTP 服务器
3. 用户切换后仅影响当前回复会话的发送配置，不修改账户长期绑定
4. 无可用服务器时下拉框禁用，并显示“请先配置 SMTP 服务器”提示
5. 发送前以当前下拉选中项解析 SMTP 参数并执行发信

**理由：**
- 右下角位置接近发送操作，降低切换成本
- 会话级切换可避免误改长期配置
- 发送前解析可保证与用户最终选择一致

### 决策 11：Mailcow 配置界面中心化重构

**选择：** 将当前单弹窗 Mailcow 配置升级为“配置中心”页面布局，采用“全局策略 + 服务器中心 + SMTP 服务器池”三层结构。

**信息架构：**
1. 顶部全局策略区：密码模式、失败策略、重试次数、超时、全局 SMTP 模板
2. 左侧 Mailcow 服务器列表：默认标记、启用状态、最近测试状态
3. 右侧服务器详情区：基础参数、SMTP 覆盖参数、单服务器测试与结果明细
4. 次级页签 SMTP 服务器池：列表化管理、默认服务器、连通性测试

**理由：**
- 当前字段密度过高，管理员配置与排障成本高
- 按服务器视角组织信息，更符合运维实际工作流
- 为后续扩展健康检查、证书与批量迁移能力预留结构

### 决策 12：服务器操作护栏与依赖拦截

**选择：** 对默认切换、禁用、删除等高风险动作增加前置校验，避免产生不可用配置。

**执行规则：**
1. 禁用默认服务器前必须先选定新的默认服务器
2. 删除服务器前必须检查绑定账户数，若存在依赖则禁止删除
3. 服务器详情中展示“影响范围”（绑定账户数）用于风险提示
4. 测试连接按“单服务器”粒度执行并保留最近一次结果摘要

**理由：**
- 防止默认服务器丢失导致新账户无法自动绑定
- 防止误删被使用中的服务器造成批量发信异常
- 将风险信息前置，降低误操作概率

## 风险 / 权衡

### 风险 1：统一密码泄露

**影响：** 所有 Mailcow 账户使用相同密码，泄露后影响全部

**缓解措施：**
- 脱敏显示，减少泄露面
- 建议使用强密码
- 定期轮换密码（需手动更新 Mailcow 账户密码）

### 风险 2：重复账户误判

**影响：** 查询 API 返回误报，导致跳过创建

**缓解措施：**
- 使用 Mailcow 官方 API `/api/v1/get/mailbox/{email}`
- 验证返回格式，处理异常情况
- 记录日志便于排查

### 风险 3：严格模式下失败阻断

**影响：** Mailcow API 不稳定时，用户无法创建邮箱账户

**缓解措施：**
- 默认宽松模式，减少影响
- 提供重试机制（已有 `mailcowRetryCount`）
- 记录失败原因，便于手动处理

### 风险 4：数据库迁移兼容性

**影响：** 旧版本数据库缺少新字段

**缓解措施：**
- 迁移脚本使用 `ALTER TABLE ADD COLUMN`，已有字段会跳过
- 新字段设置默认值，保证兼容性
- 迁移失败时记录警告，不阻断启动

### 风险 5：配置中心化改版的学习成本

**影响：** 旧管理员习惯于弹窗快速配置，新界面可能造成短期认知负担。

**缓解措施：**
- 保留高级模式 JSON 导入导出，保障专家用户效率
- 在首版提供引导文案与默认推荐配置
- 分阶段发布（先布局重构，再增加治理护栏）

## 迁移计划

### 部署步骤

1. **数据库迁移**：执行 `v3_7DB()` 添加新字段
2. **后端部署**：更新 `setting-service.js`, `mailcow-service.js`, `account-service.js`
3. **前端部署**：将 Mailcow 设置入口升级为配置中心页面，更新 `sys-setting/index.vue`
4. **灰度验证**：优先验证单服务器测试链路、默认服务器切换、依赖拦截
5. **配置验证**：管理员登录后台，完成全局策略、Mailcow 服务器池与 SMTP 服务器池校验

### 里程碑与 Go/No-Go 门禁

**M1（布局重构）**
- 交付：配置中心页面基础布局完成（全局策略、服务器列表、详情区）
- Go 条件：管理员可完成完整保存流程，旧配置可正常回显
- No-Go 条件：出现配置丢失、字段错位或无法保存

**M2（按服务器测试）**
- 交付：服务器粒度测试连接与结果展示上线
- Go 条件：测试结果与目标服务器一致，失败信息可定位
- No-Go 条件：测试结果串线、状态不一致或错误信息不可读

**M3（护栏与依赖拦截）**
- 交付：默认切换、禁用、删除依赖拦截规则生效
- Go 条件：高风险操作均被拦截并给出明确提示
- No-Go 条件：存在“无默认服务器”或“删除后悬空绑定”路径

**M4（高级模式与回归）**
- 交付：JSON 导入导出、互转一致性与核心回归完成
- Go 条件：核心路径回归通过（配置、测试、默认切换、依赖拦截）
- No-Go 条件：普通/高级模式互转损坏配置或回归失败

### 回滚策略

1. **后端回滚**：恢复旧版本代码
2. **前端回滚**：恢复旧版本 UI
3. **数据库回滚**：无需操作（新字段有默认值，旧代码忽略）

### 回滚触发条件

- 配置中心页面出现“保存成功但实际未持久化”
- 默认服务器规则失效，导致新账户无法自动绑定服务器
- 删除/禁用后出现大量账户悬空绑定且无可用修复路径
- 单服务器测试结果与实际请求目标不一致

### 数据兼容性

- 新字段有默认值，旧代码可正常运行
- 旧数据无 `mailcowPasswordMode`，默认 `random`（兼容现有行为）
- 旧数据无 `mailcowCreateStrict`，默认 `0`（宽松模式）
