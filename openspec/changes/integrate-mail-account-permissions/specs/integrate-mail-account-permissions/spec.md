## 新增需求

### 需求:邮件账户功能动态展示
系统必须允许管理员通过权限配置来控制“邮件账户”菜单在侧边栏的显示。

#### 场景:用户拥有查看账户权限
- **当** 用户角色包含 `account:query` 权限时
- **那么** 侧边栏必须显示“邮件账户”菜单项，且点击可进入 `/account` 页面

#### 场景:用户没有查看账户权限
- **当** 用户角色不包含 `account:query` 权限时
- **那么** 侧边栏禁止显示“邮件账户”菜单项，且直接访问 `/account` URL 必须被重定向或显示 404/403

### 需求:账户操作按钮级权限控制
系统必须支持对“添加账户”和“删除账户”操作进行独立的权限控制。

#### 场景:用户拥有添加权限但无删除权限
- **当** 用户仅有 `account:add` 而无 `account:delete` 权限时
- **那么** 账户列表页必须显示“添加”按钮，但每行操作区禁止显示“删除”按钮

### 需求:权限树国际化支持
系统的角色权限配置树必须能正确翻译“邮件账户”及其子操作的名称。

#### 场景:管理员配置中文环境权限
- **当** 管理员在中文界面打开角色权限配置时
- **那么** 权限树中必须显示“邮件账户”、“账户查看”、“账户添加”、“账户删除”等中文名称，而不是原始 key

### 需求:后端 API 鉴权一致性
后端 API 必须严格校验请求用户的 `permKey`，确保 UI 层面的限制无法被 API 直接调用绕过。

#### 场景:无权限用户尝试调用添加接口
- **当** 用户不具备 `account:add` 权限但尝试 POST `/account/add` 时
- **那么** 后端必须返回 403 未授权错误
